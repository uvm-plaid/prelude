\section{Abstract}
Multi-Party Computation protocols are distributed protocols where homomorphic properties are utilized to perform operations on the data while preserving its privacy. Since the protocols are distributed, the encryption of the data must protect one party's inputs from the risk that another computing party is an adversary. For example, some protocols like Goldreich-Micali-Wigderson (GMW) use additive secret shares to represent input values. GMW uses these additive secret shares to privately compute boolean circuits. Correctness properties of these protocols can be verified either manually or automatically to ensure that the output meets the expectations of the system for a given set of inputs. The \textit{Prelude} language allows for correctness properties to be represented as pre and postconditions that leverage a Satisfiability Modulo Theory solver for automatic verification. As the protocols scale in size, it becomes important to evaluate the verification time of the automatic verification tool. In this thesis, we parsed pre-defined boolean circuits for 64-bit addition and subtraction into hundreds of gates of \textit{Prelude} code, as well as 64-bit multiplication into thousands of gates, and empirically evaluated the time complexity of each circuit's evaluation. We found that 64-bit addition and 64-bit subtraction can be verified reasonably quickly, but that the size of the 64-bit multiplication circuit posed a significant challenge for the tool.

\subsection{Multi-Party Computation}
The motivation behind Multi-Party Computation (MPC) protocols at a high level is to provide a way for multiple parties to compute some shared function on their input secrets without revealing any additional information about their inputs \cite{evans2018pragmatic}. The security of a protocol and each party's input values is defined by the real-ideal paradigm. In the ideal world, there is a trusted third-party referred to as the oracle. Since each party fully trusts the oracle, they send it their plain input secrets and the oracle computes the ideal functionality on them. After computing the functionality, the oracle then shares this result with everyone. In some real world scenarios, like voting in an election, we have to trust a third-party to perform computations on the data. However, in developing MPC protocols, we assume that such an oracle does not exist. The oracle and the ideal world are set up as a way to define what information is intended to be revealed and reason about if a protocol is secure. We do this by showing that a real world protocol does not reveal any more information to a corrupt party than the ideal functionality would.

% It is also important to realize that security does not equal safety, and sometimes the ideal functionality might reveal 
%In some instances of computation, there are multiple parties wanting to contribute inputs to the same function, which will give them a shared result without revealing any information about their input secrets to any other party. 
To start with a simple example, we can look at the Millionaire's Problem. Two millionaires wish to compare their wealth and determine who has more money without revealing how much they have to the other person. The ideal functionality in this case, would compare the two values and reveal some indicator, possibly a boolean value, determining if millionaire one has more money than millionaire two or not. If we want to have a secure real world protocol to perform the comparison, then it must only reveal the fact that one millionaire has more money and nothing more. Say that millionaire two, with a balance of five million dollars, is corrupt and wants to learn additional information about the bank account of millionaire one. If the output of the function is that millionaire two does not have more money than millionaire one, the corrupt party now knows with certainty that millionaire one has at least five million and one dollars. But, that must be all that they learn. Each possible dollar amount above five million should still be equally probable. Note that some information should be leaked by the intended functionality, otherwise these protocols would have no use in the real world. MPC protocols give us a way to do this by limiting the amount of information learned by corrupt parties conditional on the output of the intended functionality.
% \todo{conclude/transition?}

MPC protocols are distributed protocols, so the parties involved may each be responsible for computing parts of the shared function and communicating information back and forth when necessary. To ensure that the communication and computation of another party's data does not reveal information about the input secrets, MPC protocols use homomorphic encryption schemes. Homomorphic encryption ensures that performing operations on encrypted values and decrypting the result has the same effect as performing those operations on the unencrypted values. For example, if we have a protocol for summing the values of three different parties together, we can generate additive secret shares $m_1$, $m_2$, and $m_3$ of each secret $S_i$ such that $S_i = m_1 + m_2 + m_3$ and that any two of the shares appear to be from a uniformly random distribution. These shares can be distributed, one to each party, who can then perform addition on their three shares and broadcast that partial sum without revealing any information about the inputs. Note that because of the uniform randomness, if any two parties are corrupt and reveal their set of three shares, it also reveals nothing about the secrets. Protocols like 3-Party Addition have correctness properties, statements about what equivalences should hold at the end of execution. For 3-Party Addition, it should be the case that the protocol output should be equivalent to the sum of the three party inputs $S_1 + S_2 + S_3$, and more generally, the output of the ideal functionality. Recent work with the \textit{Prelude} tool has explored a way to automate the verification of such correctness properties.

\subsection{Background and Related Work}
The work in this thesis is based upon the \textit{Overture/Prelude} tool first introduced in \cite{skalka-near-ppdp24}. This initial paper defines \textit{Overture}, the low-level language used for defining MPC protocols, as well as the metalanguage \textit{Prelude}, by which we mean running a \textit{Prelude} program generates \textit{Overture} programs which are the protocols. In addition, it proposed a brute-force automation technique for verifying protocols in the binary field. Since this technique required computing a truth table for all possible sets of inputs, it does not scale efficiently to larger protocols. The idea of automatic verification was extended to arbitrary prime fields in \cite{skalka-near-esop25}, when Satisfiability Modulo Finite Fields theory was used as the constraint programming technique. A key goal of the extension was to limit the amount of Satisfiability Modulo Theory (SMT) solver computation during the verification of a protocol. By using pre and postconditions to represent constraints that hold throughout these protocols, they can be verified once and assumed to hold in any instantiation in a larger circuit. This is particularly beneficial in the circuits evaluated in this thesis, as they contain hundreds to thousands of such gates. 

This thesis is based upon an even newer version of the \textit{Prelude} tool with an updated implementation capable of handling larger circuits. The verification examples from existing work focus on the Finite Fields subset of cvc5 SMT, but this thesis will work on bit-vector representations of inputs with the Bit-Vector theory \hl{add cite}. This means that pre and postcondition assertions can be made on larger numbers of bits. We will see examples of protocols with 64-bit input secrets.

For our secret sharing scheme, we will continue to use the Goldreich-Micali-Wigderson (GMW) protocol, defined for binary circuits. In GMW, input secrets are represented by secret shares obtained by xorring random values with the secrets, which reconstruct via addition (in the binary case this is equivalent to $\oplus$) to the original values \cite{evans2018pragmatic}. We turn to Bristol Circuits \cite{bristol}, a standard format for boolean circuits, for examples to test \textit{Prelude}'s automatic verification using the Bit-Vector library. This includes the definition of circuits for 64-bit addition, subtraction, and multiplication. Examples of evaluating these circuits with the GMW protocol will be presented in \textbf{\hyperref[sec:Prelude]{\textit{Prelude} Language}} and \textbf{\hyperref[sec:Methods]{Methods}}, written in the \textit{Prelude} language.

\section{\textit{Overture} and 3-Party Addition}
With the context of the 3-Party Addition example, we will briefly overview what a simple \textit{Overture} MPC protocol and its correctness property look like. MPC protocols involve secret inputs, uniform randomness to protect the privacy of each input, and messages sent between parties during the protocol. These are all present in \textit{Overture}. The secret inputs of each party are denoted as \texttt{s[$x$]} and they exist in an arbitrary finite field. Random values are represented as \texttt{r[$x$]} and they also exist in the arbitrary finite field. The most relevant variable type for the examples used in this thesis is the message send syntax, \texttt{m[$x$]}. More specifically, $\elab{\mesg{x}}{2} := \elab{a}{1}$
states that expression $a$ is evaluated by party $1$ and sent to party $2$ as message $\mesg{x}$. This makes it possible to express communications between these distributed parties. In MPC protocols information is local to specific parties (expressed with $\elab{\mesg{x}}{i}$ representing a message local to party $i$), and if another party needs a share to perform computation then it must be messaged from the party that does have it. Understanding this \textit{Overture} syntax, particularly the message sending, will be useful in reading the GMW-style \textit{Prelude} libraries used for the protocols verified in \textbf{\hyperref[sec:Prelude]{\textit{Prelude} Language}} and \textbf{\hyperref[sec:Methods]{Methods}}. 

In the 3-Party Addition example shown in in Figure \ref{fig-3-party-add}, each of the three parties generate secret shares with random samples such that any two of the shares appears uniformly random. These shares are made up of a random value that the party keeps, a random value that one of the other parties gets, and the secret minus both random values that is sent to the remaining party. It must be the case that the secret shares of each secret sum up to the secret itself. For example, $$\elab{\locflip}{1} + \elab{\mesg{s1}}{2} + \elab{\mesg{s1}}{3} \eop \elab{\secret{1}}{1}$$

One additional part of the \textit{Overture} syntax is the public reveal, expressed as \texttt{p[$x$]}. The first three lines in the right column express the summing each party does with secret shares of each of the inputs. The reveal acts as a broadcast, sending the result to the other parties. This shows the additive homomorphism at work. Since each party sums up their three of the nine secret shares and reveals them, we can sum these three values together to get the sum of the input secrets themselves. Note that this is possible since each set of secret shares sums up to their respective secret and due to the commutative and associative properties of addition, we can sum these shares up in any order. It will still compute the ideal functionality, that is $$\rvl{1} + \rvl{2} + \rvl{3} \eop \elab{\secret{1}}{1} + \elab{\secret{2}}{2} + \elab{\secret{3}}{3}$$ This leads into the last important piece of this 3-Party Addition protocol, the correctness property. The ideal functionality should reveal the summation of the three secret inputs, and we can write the real world protocol's interpretation as follows:
{$$\out{1} \eop \sx{1}{1} \fplus \sx{2}{2} \fplus \sx{3}{3}\strut$$}

This constraint makes up the postcondition of our 3-Party Addition protocol, defining the correctness property to be verified. This format is able to be translated into the constraints the cvc5 SMT solver needs to automatically verify it.

\section{Satisfiability Modulo Bit-Vector Theory}
Satisfiability Modulo Theories (SMT) are an extension of the Boolean Satisfiability problem, which searches for a set of input values that satisfy a formula composed of boolean operations. SMT generalizes this to first-order logic formulas, specific to some theory. As mentioned above, prior work used Satisfiabilty Modulo Finite Fields theory. For this thesis, we are working with Satisfiability Modulo Bit-Vector theory, meaning that we hope to search across the space of bit-vectors to determine if our set of constraints can be satisfied. We leverage the Bit-Vector theory from the Cooperating Validity Checker (cvc5) \todo{cite} SMT solver library, where all relevant operations on bit-vectors (addition, subtraction, multiplication, and concatenation) are defined. Since these functions are native to the cvc5 library, we avoid the bottleneck of making external function calls when trying to automatically verify a circuit's correctness property. 

For a trivial example, we can use cvc5's Pythonic API to establish constraints in a small system, as shown in Figure \ref{fig-cvc5}. 
\begin{figure}
{\footnotesize
\begin{center}
    \begin{tabular}{l}
    from cvc5.pythonic import *\\
    a, b = BitVecs('a b', 5)\\
    x = BitVecVal(10, 5)\\
    y = BitVecVal(22, 5)\\
    z = BitVecVal(2, 5)\\
    // Bit-Vector representation of solve(a + 10 == 2 * b, b + 22 == 2 * a)\\
    solve(BVAdd(a, x) == BVMult(z, b), BVAdd(b, y) == BVMult(z, a))\\
    \end{tabular}
\end{center}}
\caption{cvc5 Pythonic library example of SMT adapted from \cite{cvc5-example}}
\label{fig-cvc5}
\end{figure}
The \texttt{BitVecs('a b', 5)} syntax initializes two bit-vectors of size 5. \texttt{BitVecVal(10, 5)} initializes a bit-vector of size 5 and assigns it the value 10. \texttt{BVAdd(a, x)} performs addition on the bit-vectors \texttt{a} and \texttt{x}, and \texttt{BVMult(z, b)} multiplies the bit-vectors \texttt{z} and \texttt{b}. In addition to these operations, there is also \texttt{BVConcat(a,b)} which concatenates bit-vector \texttt{b} onto the end (least significant side) of bit-vector \texttt{a} and is also present in the circuits used in this thesis. Note that bit-vectors of size 5 can represent integers up to $2^5 - 1$ or 31. The cvc5 library handles overflow of bit-vector operations by discarding the carry bit. 

On the seventh line in Figure \ref{fig-cvc5}, the constraints of the system are passed as parameters to the \texttt{solve()} call. The two restrictions state that adding 10 to \texttt{a} must be twice as much as \texttt{b} and adding 22 to \texttt{b} must be twice \texttt{a}. Executing the script  will return the result \texttt{[a = 18, b = 14]}, a solution to the constraints, meaning that they are satisfiable.

The \textit{Overture/Prelude} framework has the ability to parse pre and postconditions of MPC protocols into similar constraints that the SMT solver can restrict the input and output space by. The definitions of these postconditions are what we test for satisfiability to automatically verify correctness. It is important to observe that the simple example returned one solution, but is not necessarily satisfiable for all combinations of \texttt{a} and \texttt{b}. In practice, we check the negation of the unsatisfiability of the constraints. If any set of inputs does not satisfy the model we constructed with the pre and postconditions, then the protocol is not correct. By negating the unsatisfiability, it will only return true if every single combination of inputs satisfies the constraints.

\section{\textit{Prelude} Language}
\label{sec:Prelude}
To introduce the \textit{Prelude} Language, we will start with a simple example. Suppose we have two parties, who have secrets $S_1$ and $S_2$, existing in \{0, 1\}. Our ideal functionality will be to compute the XOR of the negation of $S_1$ and $S_2$, i.e. $(\neg S_1) \oplus S_2$. Figure \ref{fig-2-party-not-xor} illustrates the boolean circuit diagram.
\begin{figure}
\centerline{\includegraphics[scale=.4]{2-party-not-xor.png}}
\caption{2-Party NOT XOR Circuit Diagram}
\label{fig-2-party-not-xor}
\end{figure}
This circuit is not secure, but we can replace the wire values and boolean gates according to the GMW protocol. Using additive secret sharing, we can split each party's secret into two shares that appear random but XOR together to reconstruct the secret as shown in the \texttt{RECON} function. Recall that XORing singular bit values is the same as adding them. In the following \textit{Prelude} code in Figure \ref{fig-prelude-gmw}, it is assumed that the input variables for each gate are already shared in this way, as each function definition handles a wire's value as a combination of party 1's share and party 2's share. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
exprfunctions:

RECON(x) { (m[x]@1 + m[x]@2) }

not(x){ x + 1 }

cmdfunctions:

xorgmw(z:string, x:string, y:string) {
   m[z]@1 := (m[x] + m[y])@1; 
   m[z]@2 := (m[x] + m[y])@2
} 
postcondition: (RECON(z) == RECON(x) + RECON(y))

notgmw(z:string, x:string) {
   m[z]@1 := m[x]@1;
   m[z]@2 := not(m[x])@2
}
postcondition: ( RECON(z) == not(RECON(x)) )
\end{verbatimtab}
}
\caption{\textit{Prelude} libraries for GMW-style gates}
\label{fig-prelude-gmw}
\end{figure}
Another property of operations on bits is that the negation of a value made up of shares x1 and x2, is equivalent to $x1 \oplus (\neg x2)$. This is utilized in the function definition of \texttt{notgmw(z,x)}. With these \textit{Prelude} functions, we can write the body of a main function that securely represents the circuit and realizes our ideal functionality in Figure \ref{fig-2-party-not-xor-code}. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
main() {
   notgmw("g1", "x");
   xorgmw("z", "g1", "y")
} 
postcondition: (RECON("z") == RECON("y") + not(RECON("x")))
\end{verbatimtab}
}
\caption{2-Party NOT XOR example protocol}
\label{fig-2-party-not-xor-code}
\end{figure}
We define the postcondition to assert the correctness property of the circuit. It states that the reconstruction of the output \texttt{"z"} should be equivalent to the sum ($\oplus$) of the reconstruction of input \texttt{"y"} with the negation of input \texttt{"x"}, which was our original intended functionality. This circuit verifies with an empty precondition, indicating it holds for all values of \texttt{"x"} and \texttt{"y"}. 

In collaboration with PhD Student Steven Baldasty and my advisor Dr. Christian Skalka, we also discussed and implemented changes to the \textit{Prelude} backend to include syntax for \texttt{BVAdd()}, \texttt{BVSub()}, \texttt{BVMult()}, and "\texttt{$\vert$}", the equivalent of BVConcat in the cvc5 Bit-Vector SMT solver. This new syntax allows the tool to be used to verify circuits with bit-vector representations of numbers as inputs for addition, subtraction, and multiplication. There are two reasons for pivoting away from SMFF theory to the Bit-Vector theory. We believe that this library is more optimized and that having these functions natively defined in the cvc5 Bit-Vector library, we can more efficiently call them to perform operations on these larger circuits, reducing verification time. The other advantage is that it saves us the effort of redefining the ideal functionality for each operation, which would have been tedious. These functions can be invoked in the pre and postconditions of the larger circuits we tested and described below. % should this be a \hyperref?

\section{Methods}
\label{sec:Methods}
For an introduction to combining the new bit-vector syntax in \textit{Prelude} with an MPC protocol, we started with a 2-bit adder referenced from \cite{2-bit-adder}. The 2-bit adder, as shown in Figure \ref{fig-2-bit-adder-circuit} initially contained XOR, AND, and OR boolean gates. Due to the cvc5 BVAdd definition dropping the carry bit, we were able to ignore the carry bit output wire and the corresponding gates. In addition to the \texttt{xorgmw(z,x,y)} \textit{Prelude} function defined in Figure \ref{fig-prelude-gmw}, we used additional libraries for an \texttt{andgmw(z,x,y)} function that utilizes oblivious transfer as defined in the GMW protocol. This definition has been omitted for simplicity.

\begin{figure}
    \centerline{\includegraphics[scale=0.5]{2-bit-adder.png}}
    \caption{2-Bit Adder circuit diagram adapted from \cite{2-bit-adder}}
    \label{fig-2-bit-adder-circuit}
\end{figure}
Without the carry bit output wire, there are only four relevant boolean gates to translate into \textit{Prelude} code. The main function for the 2-bit adder in Figure \ref{fig-2-bit-adder} defines the secure version of the adder.
\begin{figure}

\begin{verbatimtab}
main() {
   xorgmw("z1", "x1", "y1"); // Bit 0 (least significant)
   andgmw("g1", "x1", "y1");
   xorgmw("g2", "x2", "y2");
   xorgmw("z2", "g2", "g1") // Bit 1 (most significant)
} 
postcondition: (|RECON("z2"), RECON("z1")| == BVAdd(|RECON("x2"), RECON("x1")|,

\end{verbatimtab}
}

Note that the postcondition explicitly states the correctness property. The output bit-vector, generated from the two output bits concatenated via "$\vert$", is equivalent to the sum of the two bit-vector inputs. Given the body of main and the postcondition, we used the \textit{Prelude} tool to pass the constraints to the SMT solver and automatically verify the protocol. As a small circuit, this verified quickly.

With a successful verification of a simple circuit using the Bit-Vector theory SMT solver, we decided to expand the evaluation to larger boolean circuits, using the format of Bristol Circuits \cite{bristol}. Each line of the Bristol Circuit format includes the number of input wires, number of output wires, list of input and output wires, and the type of boolean gate operation. For the circuits we wished to implement, it was not reasonable to manually translate hundreds or thousands of these lines into \textit{Prelude} code. 

The solution was to write a parsing script in Perl. The script was made in collaboration with my advisor, Dr. Christian Skalka. Perl has well documented syntax for regular expressions, which was useful in parsing the Bristol Circuit format. For each line, we use the character class \textbackslash w to capture the gate operation. We could then go case by case for each gate and use the \textbackslash d+ character class to retrieve the input and output wire values. These values are all that is necessary to generate a \textit{Prelude} function call, which was printed to the terminal for each gate. After the initial script, additional modifications were made to include a check for XOR gates, which are present in both of the larger circuits we wanted to test the tool on. The script was also extended to automatically generate the postcondition. The postcondition reversed the order of inputs to maintain the proper bit significance used in the cvc5 library, used \texttt{BVConcat} on both inputs and outputs, and the appropriate \texttt{BVAdd/BVSub/BVMult} operation.

The parser was run on Bristol Circuit text files for the 64-bit Adder, 64-bit Subtract, and 64-bit Multiplier circuits \cite{bristol}. It was then straightforward to populate the body of the main function with the list of GMW-style gate function calls, and add the bit-vector operations to the corresponding postconditions. The implementations, ranging from hundreds to thousands of gates, are available in the \textit{Prelude} repository \cite{prelude-github}. When run with the \textit{Prelude} engine, the correctness properties for the additon and subtraction circuits are automatically verified after several seconds.

\section{Conclusions and Future Work}
From the table in \textbf{\hyperref[sec:Results]{Experiments and Results}}, we can see that the 64-Bit Addition and 64-Bit Subtraction circuits verified in a very reasonable amount of time, both under 6 seconds. It does not take significantly longer than the 2 and 4 gate circuits. The 64-Bit Multiplication circuit, however, posed a challenge for the tool. Due to the larger circuit size of nearly 14,000 gates, the tool ran into an issue generating contracts for each of the constraints to be modeled with the cvc5 solver. When running this circuit on a personal MacBook Air with the M2 chip, we ran into an OutOfMemoryError for Java heap space. PhD Student Steven Baldasty ran the circuit on the Vermont Advanced Computing Center (VACC) machine and was able to allocate enough space to expand the function definitions from \textit{Prelude}, but it still got stuck when trying to turn the contracts into constraints for the cvc5 solver. This was with 90 gigabytes of memory allocated to the task. 

Future work related to this thesis could investigate possible optimizations that can be made within the translation process for cvc5 constraints to more efficiently use memory and avoid the problem of running out of heap space. Currently, memory for the constraints is the bottleneck and we are unable to test if the verificaition of larger circuits would complete in a reasonable amount of time. If it is possible to refine the process for translating the function definitions into cvc5 constraints, the Bristol Circuits also have a 64-Bit Signed Division circuit that contains nearly 30,000 gates and can serve as the next step up in scalability testing for automatic verification.
