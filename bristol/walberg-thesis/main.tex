\section{Abstract}
Multi-Party Computation protocols are distributed protocols where homomorphic properties are utilized to perform operations on the data while preserving its privacy. Since the protocols are distributed, the encryption of the data must protect one party's inputs from the risk that another computing party is an adversary. For example, some protocols like Goldreich-Micali-Wigderson use additive secret shares to represent input values. These can be used for simple summation protocols where addition can be performed locally using the additive homomorphism. Correctness properties of these protocols can be verified either manually or automatically to ensure that the output meets the expectations of the system for a given set of inputs. The \textit{Prelude} language allows for correctness properties to be represented as pre and postconditions that leverage a Satisfiability Modulo Theory solver for automatic verification. As the protocols scale in size, it becomes important to evaluate the verification time of the automatic verification tool. In this thesis, we parse Boolean circuits for 64-bit addition and subtraction into \textit{Prelude} code, resulting in hundreds of gates, whose verifications are timed and compared against a smaller 2-bit addition circuit.

\section{Introduction}
\subsection{Multi-Party Computation}
The motivation behind Multi-Party Computation (MPC) protocols at a high level is to provide a way for multiple parties to compute some shared function on their input secrets without revealing any additional information about their inputs \cite{evans2018pragmatic}. The security of a protocol and each party's input values is defined by the real-ideal paradigm. In the ideal world, there is a third-party referred to as the functionality, which every other party completely trusts to perform the shared function on their inputs. Each party shares their inputs with the functionality, which then performs the function directly on the inputs. After computing the function, the third-party then shares this result with everyone. In practice, of course, a completely trusted functionality does not exist. However, it is a way to measure the security of a given protocol and distinguish between what information is intended to be revealed and what an adversary might learn during execution. The way we determine if a protocol in the real world is secure is by showing that it doesn't reveal any more information to a corrupt party than the ideal functionality would.

%In some instances of computation, there are multiple parties wanting to contribute inputs to the same function, which will give them a shared result without revealing any information about their input secrets to any other party. 
To start with a simple example, we can look at the Millionaire's Problem. Two millionaires wish to compare their wealth and determine who has more money without revealing how much they have to the other person. The ideal functionality in this case, would compare the two values and reveal some indicator, possibly a boolean value, determining if millionaire one has more money than millionaire two or not. If we want to have a secure real world protocol to perform the balance comparison, then it must only reveal the fact that one millionaire has more money and nothing more. Let's say that millionaire two, with a balance of five million dollars, is corrupt and wants to learn additional information about the bank account of millionaire one. If the output of the function is that millionaire two does not have more money than millionaire one, the corrupt party now knows with certainty that millionaire one has at least five million and one dollar. But, that must be all that they learn. Each possible dollar amount above five million should still be equally probable. Note that some information should be leaked by the intended functionality, otherwise these protocols would have no use in the real world. MPC protocols give us a way to do this by limiting the amount of information learned by corrupt parties conditional on the output of the intended functionality.
% \todo{conclude/transition?}

MPC protocols are distributed protocols, so the parties involved may each be responsible for computing parts of the shared function and communicating information back and forth when necessary. To ensure that the communication and computation of another party's data does not reveal information about the input secrets,\hl{ MPC protocols use homomorphic encryption schemes. Homomorphic encryption allows for computation to be performed on encrypted values } that would be equivalent to the computation on the plain values. For example, if we have a protocol for summing the values of three different parties together, we can generate additive secret shares of each secret $S_i$ by taking two random values $m_1, m_2$ and then a third value $m_3$ such that $S_i = m_1 + m_2 + m_3$. These shares can be distributed one to each party who can then perform addition on their three shares and broadcast that partial sum without revealing any information about the inputs. Protocols like 3-Party Addition have correctness properties, statements about what equivalences should hold at the end of execution. For 3-Party Addition, it should be the case that the output should be equivalent to the sum of the three party inputs $S_1 + S_2 + S_3$. Recent work with the \textit{Prelude} tool has explored a way to automate the verification of such correctness properties.

\subsection{Background and Related Work}
The work in this thesis is based upon the \textit{Prelude/Overture} tool first introduced in \cite{skalka-near-ppdp24}. This initial paper defines \textit{Overture}, the low-level language used for defining MPC protocols as well as \textit{Prelude}, the metalanguage used to dynamically create \textit{Overture} protocols. In addition, it proposed a brute-force automation technique for verifying protocols in the binary field. Since this technique required computing a truth table for all possible sets of inputs, it does not scale efficiently to larger protocols. The idea of automatic verification was extended to arbitrary prime fields in \cite{skalka-near-esop25}, when Satisfiability Modulo Finite Fields theory was used as the constraint programming technique. A key goal of the extension was to limit the amount of SMT solver computation during the verification of a protocol. By using pre and post conditions to represent constraints that hold throughout these protocols, they can be verified once and assumed to hold in any instantiation in a larger circuit. This is particularly beneficial in the circuits evaluated in this thesis, as they contain hundreds of such gates. 

This thesis is based upon an even newer version of the \textit{Prelude} tool with an updated implementation capable of handling larger circuits. The verification examples from existing work focus on the Finite Fields subset of cvc5 SMT, but this thesis will work on bit-vector representations of inputs with the Bit-Vector theory. This means that pre and postcondition assertions can be made on larger numbers of bits. We will see examples of protocols with 64 bit input secrets.

For our secret sharing scheme, we will continue to use the Goldreich-Micali-Wigderson (GMW) protocol, working with bits. In GMW, input secrets are represented by secret shares padded with random values, which reconstruct via addition (in the binary case this is equivalent to $\oplus$) to the original values \cite{evans2018pragmatic}. We turn to Bristol Circuits \cite{bristol}, a standard format for boolean logic circuits, for examples to test \textit{Prelude}'s automatic verification using the Bit-Vector library. Examples of evaluating these boolean circuits with the GMW protocol will be presented in \textbf{\hyperref[sec:Prelude]{\textit{Prelude} Language}} and \textbf{\hyperref[sec:Methods]{Methods}}, written in the \textit{Prelude} language.

\section{\textit{Overture} and 3-Party Addition}
We can return to the example of 3-Party Addition to briefly overview what a simple \textit{Overture} MPC protocol and its correctness property look like. In this protocol, parties $P_1, P_2,$ and $P_3$ have secrets $S_i$ such that $S_i = m_1 + m_2 + m_3$ where each secret share $m_j$ reveals no information about the secret due to a random one-time pad. The parties wish to compute $\sum S_i$ and can do so securely with the \textit{Overture} protocol defined in figure \ref{fig-3-party-add}.

Using the message notation $\elab{\mesg{si}}{j} := \elab{(\secret{i} \fminus \locflip \fminus \flip{x})}{i}$ each party i is sharing one secret share of their secret $si$ with each other party j. Since these shares are based on random one-time pads, it does not matter that another party sees the value, as it reveals nothing about the input secret. Now each party has one share of their own secret, as well as one share of each of the other two secrets. Each party can locally compute the sum of their shares and broadcast it to the other parties without leaking any information, with the public reveal notation $\rvl{1} := \elab{(\locflip \fplus \mesg{s2} \fplus \mesg{s3})}{1}$. 

Due to the additive homomorphism of F2 (\hl{Should this be explained?}), which ensures computations on secret shares are equivalent to computations on the underlying values, the parties can then add these partial sums to find the sum of the three input secrets without any party learning information about another party's secret. Understanding this \textit{Overture} syntax, particularly the message sending, will be useful in reading the GMW-style \textit{Prelude} libraries used for the protocols verified in \textbf{\hyperref[sec:Prelude]{\textit{Prelude} Language}} and \textbf{\hyperref[sec:Methods]{Methods}}. 

The last important piece of this 3-Party Addition protocol, is the correctness property. The ideal functionality should reveal the summation of the three secret inputs, and we can write the real world protocol's interpretation as follows:

{$$\out{1} \eop \sx{1}{1} \fplus \sx{2}{2} \fplus \sx{3}{3}\strut$$}

This constraint makes up the postcondition of our 3-Party Addition protocol, defining the correctness property to be verified. This format is easily translated into the constraint the cvc5 SMT library needs to automatically verify it.

We can evaluate what a corrupt party can observe to determine if any additional information is learned. This assumes a semi-honest adversarial threat model, that is, corrupt parties will try to learn additional information through observation, but will still follow the protocol. In the Millionaire's problem, the protocol reveals who has more money. For the millionaire with less money, this eliminates all possible values for the input of the richer millionaire below their own balance. From the output and their own input, they know the other millionaire must have at least as much money as they do, but each of the dollar amounts above their own is still equally likely. In 3-Party Addition, if a party is corrupt, they can subtract their secret from the output to get the sum of the other party's inputs, but learn nothing more about what they are individually. The goal of verifying the security of an MPC protocol, is that conditional on the output of $F$, each possible value is equally likely.

These constraint statements can be easily translated into constraints in a Satisfiability Modulo Theory (SMT) solver, where the protocol's correctness verification can be automated. SMT is a more general version of Boolean Satisfiability, where the solver is presented with a list of boolean constraints and iterates over all combinations of possible values to determine if there is a set of booleans that satisfies the constraints. SMT does the same thing but with arbitrarily sized integers. For our work, we use the Cooperating Validity Checker (cvc5) \todo{cite this} library to run models over the input secret space and determine if protocols hold for all combinations of values. For a trivial example, we can use cvc5's Pythonic API to establish constraints in a small system. 
\begin{figure}
{\footnotesize
\begin{center}
    \begin{tabular}{l}
    from cvc5.pythonic import *\\
    a, b = Ints('a b')\\
    solve(a + 10 == 2 * b, b + 22 == 2 * a)\\
    \end{tabular}
\end{center}}
\caption{cvc5 pythonic library example of SMT}
\label{fig-cvc5}
\end{figure}
Executing this script will return the result [a = 18, b = 14], a solution to the system of constraints defined on the third line. A solution means that the constraints are satisfiable. In practice with the larger protocols, we check the unsatisfiability of the constraints. This will return true if any set of input secrets results in an output that does not match the expectation of the postcondition. The cvc5 Bit-Vector library has native functions for adding, subtracting, and concatenating bit-vectors, which allow us to perform these operations without the costly bottleneck of making an external call to custom functions to achieve the same goal.

\section{\textit{Prelude} Language}
\label{sec:Prelude}
To introduce the \textit{Prelude} Language, we will start with a simple example. Suppose we have two parties P1, and P2, who have secrets S1 and S2 respectively, existing in \{0, 1\}. Our ideal functionality will be to compute the XOR of the negation of S1 and S2, i.e. $(\neg S1) \oplus S2$. Figure \ref{fig-2-party-not-xor} made using \todo{cite circuit drawer tool} illustrates the boolean circuit diagram.
\begin{figure}
\centerline{\includegraphics[scale=.4]{2-party-not-xor.png}}
\caption{2-Party NOT XOR Circuit Diagram}
\label{fig-2-party-not-xor}
\end{figure}
This circuit is not secure, but we can replace the wire values and boolean gates according to the GMW protocol. Using additive secret sharing, we can split each party's secret into two shares that appear random but XOR together to reconstruct the secret as shown in the RECON function. Recall that XORing singular bit values is the same as addition. In the following \textit{Prelude} code, it is assumed that the input variables for each gate are already shared in this way, as each function definition handles a wire's value as a combination of party 1's share and party 2's share. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
exprfunctions:

RECON(x) { (m[x]@1 + m[x]@2) }

not(x){ x + 1 }

cmdfunctions:

xorgmw(z:string, x:string, y:string) {
   m[z]@1 := (m[x] + m[y])@1; 
   m[z]@2 := (m[x] + m[y])@2
} 
postcondition: (RECON(z) == RECON(x) + RECON(y))

notgmw(z:string, x:string) {
   m[z]@1 := m[x]@1;
   m[z]@2 := not(m[x])@2
}
postcondition: ( RECON(z) == not(RECON(x)) )
\end{verbatimtab}
}
\caption{\textit{Prelude} libraries for GMW-style gates}
\label{fig-prelude-gmw}
\end{figure}
Another property of operations on bits is that the negation of a value made up of shares x1 and x2, is equivalent to $x1 \oplus (\neg x2)$. This is utilized in the function definition of notgmw(z,x). With these \textit{Prelude} functions, we can write the body of a main function that securely represents the circuit and realizes our ideal functionality in figure \ref{fig-2-party-not-xor-code}. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
main() {
   notgmw("g1", "x");
   xorgmw("z", "g1", "y")
} 
postcondition: (RECON("z") == RECON("y") + not(RECON("x")))
\end{verbatimtab}
}
\caption{2-Party NOT XOR Example Protocol}
\label{fig-2-party-not-xor-code}
\end{figure}
We define the postcondition to assert the correctness property of the circuit. It states that the reconstruction of the output "z" should be equivalent to the sum ($\oplus$) of the reconstruction of input "y" with the negation of input "x", which was our original intended functionality. This circuit verifies with an empty precondition, indicating it holds for all values of "x" and "y". 

In collaboration with PhD Student Steven Baldasty and my advisor Dr. Christian Skalka, we also discussed and implemented changes to the \textit{Prelude} backend to include syntax for BVAdd(), BVSub(), and "$\vert$", the equivalent of BVConcat in the cvc5 Bit-Vector SMT solver. This new functionality allows the tool to be used to verify circuits with bit-vector representations of numbers as inputs for addition and subtraction. These functions are invoked in the pre and postconditions of the larger circuits tested and described in \textbf{\hyperref[sec:Methods]{Methods}}. There is an advantage to these functions being defined in \textit{Prelude} and within the cvc5 library. We don't have to make external calls to custom function definitions and we can instead work within the optimized libraries, reducing the verification time.

\section{Methods}
\label{sec:Methods}
For an introduction to combining the new bit-vector syntax in \textit{Prelude} with an MPC protocol, we started with a 2-bit adder \hl{add CMU lab citation to bibtex}. The 2-bit adder, as shown in figure \ref{fig-2-bit-adder-circuit} initially contained XOR, AND, and OR boolean gates. Due to the cvc5 BVAdd definition dropping the carry bit, we were able to ignore the carry bit output wire and the corresponding gates. In addition to the xorgmw(z,x,y) \textit{Prelude} function defined in figure \ref{fig-prelude-gmw}, we used additional libraries for an andgmw(z,x,y) function that utilizes oblivious transfer as defined in the GMW protocol. This definition has been omitted for simplicity.

\begin{figure}
    \centerline{\includegraphics[scale=0.75]{2-bit-adder.png}}
    \caption{2-Bit Adder Circuit Diagram}
    \label{fig-2-bit-adder-circuit}
\end{figure}
Without the carry bit output wire, there are only four relevant boolean gates to translate into \textit{Prelude} code. The main function for the 2-bit adder in figure \ref{fig-2-bit-adder} defines the secure version of the adder.
\begin{figure}
{\footnotesize
\begin{verbatimtab}
main() {
   xorgmw("z1", "x1", "y1"); // Bit 0 (least significant)
   andgmw("g1", "x1", "y1");
   xorgmw("g2", "x2", "y2");
   xorgmw("z2", "g2", "g1") // Bit 1 (most significant)
} 
postcondition: (|RECON("z2"), RECON("z1")| == BVAdd(|RECON("x2"), RECON("x1")|, 
                                                   |RECON("y2"), RECON("y1")|))
\end{verbatimtab}
}
\caption{2-Bit Adder in \textit{Prelude}}
\label{fig-2-bit-adder}
\end{figure}
Note that the postcondition explicitly states the correctness property. The output bit-vector, generated from the two output bits concatenated via "$\vert$", is equivalent to the sum of the two bit-vector inputs. Given the body of main and the postcondition, we used the \textit{Prelude} tool to pass the constraints to the SMT solver and automatically verify the protocol. As a small circuit, this verified quickly.

With a successful verification of a simple circuit using the bit-vector theory SMT solver, we decided to expand the evaluation to larger boolean circuits, using the format of Bristol Circuits \cite{bristol}. As described in the \textbf{\hyperref[sec:Contributions]{Contributions}},\todo{Change to its own intro subsection} each line of the Bristol Circuit format includes the number of input wires, number of output wires, list of input and output wires, and the type of boolean gate operation. For the circuits we wished to implement, it was not reasonable to manually translate hundreds of these lines into \textit{Prelude} code. 

The solution was to write a parsing script in Perl. The script was made in collaboration with my advisor, Dr. Christian Skalka. Perl has well documented syntax for regular expressions, which was useful in parsing the Bristol Circuit format. For each line, we use the character class \textbackslash w to capture the gate operation. We could then go case by case for each gate and use the \textbackslash d+ character class to retrieve the input and output wire values. These values are all that is necessary to generate a \textit{Prelude} function call, which was printed to the terminal for each gate. After the initial script, additional modifications were made to include a check for XOR gates, which are present in both of the larger circuits we wanted to test the tool on. The script was also extended to automatically generate the post condition. The post condition reversed the order of inputs to maintain the proper bit significance used in the cvc5 library, used BVConcat on both inputs and outputs, and the appropriate BVAdd/BVSub operation.

The parser was run on Bristol Circuit text files for the 64-bit Adder and 64-bit Subtract circuits \cite{bristol}. It was then straightforward to populate the body of the main function with the list of GMW-style gate function calls, and add the BVAdd()/BVSub() operations to the corresponding postconditions. The implementations are available in the \textit{Prelude} repository \cite{prelude-github}, containing hundreds of gates. When run with the \textit{Prelude} engine, the correctness properties are automatically verified after several seconds.
