\section{Abstract}
Multi-Party Computation protocols are distributed protocols where homomorphic properties are utilized to perform operations on the data while preserving its privacy. Since the protocols are distributed, the encryption of the data must protect one party's inputs from the risk that another computing party is an adversary. For example, some protocols like Goldreich-Micali-Wigderson (GMW) use additive secret shares to represent input values. GMW uses these additive secret shares to privately compute Boolean circuits. Correctness properties of these protocols can be verified either manually or automatically to ensure that the output meets the expectations of the system for a given set of inputs. The \textit{Prelude} language allows for correctness properties to be represented as pre and postconditions that leverage a Satisfiability Modulo Theory solver for automatic verification. As the protocols scale in size, it becomes important to evaluate the verification time of the automatic verification tool. In this thesis, we parsed pre-defined Boolean circuits for 64-bit addition and subtraction into hundreds of gates of \textit{Prelude} code, as well as 64-bit multiplication into thousands of gates, and empirically evaluated the time complexity of each circuit's evaluation. We found that 64-bit addition and 64-bit subtraction can be verified reasonably quickly, but that the size of the 64-bit multiplication circuit posed a significant challenge for the tool.

\section{Introduction}
We will start by introducing Multi-Party Computation and provide context with a simple example. We will also cover existing work regarding the \textit{Overture/Prelude} framework and how this thesis contributes to an improved understanding of how it performs when circuit size grows.
\subsection{Multi-Party Computation}
The motivation behind Multi-Party Computation (MPC) protocols at a high level is to provide a way for multiple parties to compute some shared function on their input secrets without revealing any additional information about their inputs \cite{evans2018pragmatic}. The security of a protocol and each party's input values is defined by the real-ideal paradigm. In the ideal world, there is a trusted third-party referred to as the oracle. Since each party fully trusts the oracle, they send it their plain input secrets, and the oracle computes the ideal functionality on them. After computing the functionality, the oracle then shares this result with everyone. In some real world scenarios, like voting in an election, we have to trust a third-party to perform computations on the data. However, in developing MPC protocols, we assume that such an oracle does not exist. The oracle and the ideal world are set up as a way to define what information is intended to be revealed and to reason about whether a protocol is secure. We do this by showing that a real world protocol does not reveal any more information to a corrupt party than the ideal functionality would.

To start with a simple example, we can look at the Millionaire's Problem. Two millionaires wish to compare their wealth and determine who has more money without revealing how much they have to the other person. The ideal functionality in this case would compare the two values and reveal some indicator, possibly a Boolean value, determining if millionaire one has more money than millionaire two or not. If we want to have a secure real world protocol to perform the comparison, then it must only reveal the fact that one millionaire has more money and nothing more. Say that millionaire two, with a balance of five million dollars, is corrupt and wants to learn additional information about the bank account of millionaire one. If the output of the function is that millionaire two does not have more money than millionaire one, the corrupt party now knows with certainty that millionaire one has at least five million and one dollars. But that must be all that they learn. Each possible dollar amount above five million should still be equally probable. Note that some information should be leaked by the intended functionality, otherwise these protocols would have no use in the real world. MPC protocols give us a way to do this by limiting the amount of information learned by corrupt parties, conditional on the output of the intended functionality.

MPC protocols are distributed protocols, so the parties involved may each be responsible for computing parts of the shared function and communicating information back and forth when necessary. To ensure that the communication and computation of another party's data does not reveal information about the input secrets, MPC protocols use homomorphic encryption schemes. Homomorphic encryption ensures that performing operations on encrypted values and decrypting the result has the same effect as performing those operations on the unencrypted values. For example, if we have a protocol for summing the values of three different parties together, we can generate additive secret shares $m_1$, $m_2$, and $m_3$ of each secret $S_i$ such that $S_i = m_1 + m_2 + m_3$ and that any two of the shares appear to be from a uniformly random distribution. These shares can be distributed, one to each party, who can then perform addition on their three shares and broadcast that partial sum without revealing any information about the inputs. Note that because of the uniform randomness, if any two parties are corrupt and reveal their set of three shares, it also reveals nothing about the secrets. Protocols like 3-Party Addition have correctness properties, statements about what equivalences should hold at the end of execution. For 3-Party Addition, it should be the case that the protocol output should be equivalent to the sum of the three party inputs $S_1 + S_2 + S_3$, and more generally, the output of the ideal functionality. Recent work with the \textit{Prelude} tool has explored a way to automate the verification of such correctness properties.

\subsection{Background and Related Work}
The work in this thesis is based upon the \textit{Overture/Prelude} tool first introduced in \cite{skalka-near-ppdp24}. This initial paper defines \textit{Overture}, the low-level language used for defining MPC protocols, as well as the metalanguage \textit{Prelude}, by which we mean running a \textit{Prelude} program generates \textit{Overture} programs, which are the protocols. In addition, it proposed a brute-force automation technique for verifying protocols in the binary field $\mathbb{F}_2$. Since this technique required computing a truth table for all possible sets of inputs, it does not scale efficiently to larger protocols. The idea of automatic verification was extended to arbitrary prime fields in \cite{skalka-near-esop25}, when Satisfiability Modulo Finite Fields theory was used as the constraint programming technique. A key goal of the extension was to limit the amount of Satisfiability Modulo Theory (SMT) solver computation during the verification of a protocol. By using pre and postconditions to represent constraints that hold throughout these protocols, they can be verified once and assumed to hold in any instantiation in a larger circuit. This is particularly beneficial in the circuits evaluated in this thesis, as they contain hundreds to thousands of such gates. 

This thesis is based upon an even newer version of the \textit{Prelude} tool with an updated implementation capable of handling larger circuits. The verification examples from existing work focus on the Finite Fields subset of cvc5 SMT, but this thesis will work on bit-vector representations of inputs with the Bit-Vector theory. This means that pre and postcondition assertions can be made on larger numbers of bits. We will see examples of protocols with 64-bit input secrets.

For our secret sharing scheme, we will use the Goldreich-Micali-Wigderson (GMW) protocol, previously considered in \cite{skalka-near-ppdp24, skalka-near-esop25}. GMW is defined for binary circuits. Input secrets are represented by secret shares obtained by xoring uniformly random values with the secrets. These shares reconstruct via addition (in the binary case, this is equivalent to xor) to the original values \cite{evans2018pragmatic}. We turn to Bristol Circuits \cite{bristol}, a standard format for Boolean circuits, for examples to test \textit{Prelude}'s automatic verification using the Bit-Vector library. This includes the definition of circuits for 64-bit addition, subtraction, and multiplication. Examples of evaluating these circuits with the GMW protocol will be presented in Sections \ref{sec:Prelude} and \ref{sec:Methods}, written in the \textit{Prelude} language.

\subsection{Contributions}
\label{sec:Contributions}
Previous work related to \textit{Prelude}'s automatic verification tool \cite{skalka-near-esop25} evaluated its performance on smaller circuits with arbitrary finite fields using the Satisfiability Modulo Finite Fields (SMFF) theory, but this thesis will extend the evaluation of the scalability of \textit{Prelude} as a verification engine to larger circuits using Satisfiability Modulo Bit-Vector theory. We believe that the Bit-Vector theory provides a more optimized way to verify circuits since the relevant addition, subtraction, multiplication, and concatenation operations for bit-vectors are predefined and native to the library. Using the cvc5 Bit-Vector library was also a way to introduce the ideal functionality for each operation into the language without having to define them ourselves. 

In collaboration with PhD Student Steven Baldasty and my advisor Dr. Christian Skalka, the language's syntax was expanded to include \texttt{BVAdd()}, \texttt{BVSub()}, \texttt{BVMult()}, and "\texttt{|}" (representing \texttt{BVConcat()}) operations on bit-vectors. These functions allow us to add, subtract, multiply, and concatenate bit-vectors in the pre and postconditions of our protocols using the same behavior as the cvc5 Bit-Vector library.

In further collaboration with Dr.\ Christian Skalka, a Perl parsing script was modified to translate Bristol Circuits into \textit{Prelude} MPC protocols. These circuits are large, and it is not reasonable to implement them manually. The Perl script automatically generates postconditions for verifying the correctness properties of the circuits, which is also a tedious manual task. From the parser, we contribute code that can be used to perform the automatic verification of 64-bit addition, 64-bit subtraction, and 64-bit multiplication Boolean circuits.

\section{\textit{Overture} and 3-Party Addition}
With the context of the 3-Party Addition example, we will briefly overview what a simple \textit{Overture} MPC protocol and its correctness property look like. Among other things, MPC protocols involve secret inputs, uniform randomness to protect the privacy of each input, and messages sent between parties during the protocol. These are all present in \textit{Overture}. In a previous paper \cite{skalka-near-ppdp24}, these values existed in the binary finite field $\mathbb{F}_2$, and in \cite{skalka-near-esop25} this was extended to arbitrary finite fields. In this thesis, we will focus on the binary field $\mathbb{F}_2$. Boolean algebra can be defined in $\mathbb{F}_2$, where the \textit{xor} operation is equivalent to $+$ and the \textit{and} operation is equivalent to $*$. We will use these interchangeably for the remainder of this thesis. True and false are represented as 1 and 0, respectively, and negation will be performed by adding (xoring) 1 to the value. Finally, since the values within the field are finite, random sampling is truly uniform since we can assign each value an equal probability. 

In \textit{Overture}, the secret inputs of each party are denoted as \texttt{s[$x$]}. Random values are sampled uniformly from the field and are represented as \texttt{r[$x$]}. The most relevant variable type for the examples used in this thesis is the message send syntax, \texttt{m[$x$]}. More specifically, $\elab{\mesg{x}}{2} := \elab{\varepsilon}{1}$
states that expression $\varepsilon$ is evaluated by party $1$ and sent to party $2$ as message $\mesg{x}$. This makes it possible to express communications between these distributed parties. In MPC protocols, information is local to specific parties (expressed with $\elab{\mesg{x}}{i}$ representing a message local to party $i$), and if another party needs a share to perform computation, then it must be messaged from the party that does have it. Understanding this \textit{Overture} syntax, particularly the message sending, will be useful in reading the GMW-style \textit{Prelude} libraries used for the protocols verified in Sections \ref{sec:Prelude} and \ref{sec:Methods}. 

In the 3-Party Addition example shown in Figure \ref{fig-3-party-add}, each of the three parties generate secret shares with random samples such that any two of the shares appear uniformly random. These shares are made up of a random value that the party keeps, a random value that one of the other parties gets, and the secret minus both random values that is sent to the remaining party. It must be the case that the secret shares of each secret sum up to the secret itself. For example, $$\elab{\locflip}{1} + \elab{\mesg{s1}}{2} + \elab{\mesg{s1}}{3} \eop \elab{\secret{1}}{1}$$

One additional part of the \textit{Overture} syntax is the public reveal, expressed as \texttt{p[$x$]}. Lines 7 through 9 express the summing each party does with secret shares of each of the inputs. The reveal acts as a broadcast, sending the result to the other parties. This shows the additive homomorphism at work. Since each party sums up their three of the nine secret shares and reveals them, we can sum these three values together to get the sum of the input secrets themselves. Note that this is possible since each set of secret shares sums up to their respective secret, and due to the commutative and associative properties of addition, we can sum these shares up in any order. It will still compute the ideal functionality, that is $$\rvl{1} + \rvl{2} + \rvl{3} \eop \elab{\secret{1}}{1} + \elab{\secret{2}}{2} + \elab{\secret{3}}{3}$$ This leads into the last important piece of this 3-Party Addition protocol, the correctness property. The ideal functionality should reveal the summation of the three secret inputs, and we can write the real world protocol's interpretation as follows:
{$$\out{1} \eop \sx{1}{1} \fplus \sx{2}{2} \fplus \sx{3}{3}\strut$$}

This constraint makes up the postcondition of our 3-Party Addition protocol, defining the correctness property to be verified. This format is able to be translated into the constraints the cvc5 SMT solver needs to automatically verify it.

\section{Satisfiability Modulo Bit-Vector Theory}
Satisfiability Modulo Theories (SMT) are an extension of the Boolean Satisfiability problem, which searches for a set of input values that satisfy a formula composed of Boolean operations. SMT generalizes this to first-order logic formulas, specific to some theory. As mentioned above, prior work used Satisfiability Modulo Finite Fields theory. For this thesis, we are working with Satisfiability Modulo Bit-Vector theory, meaning that we hope to search across the space of bit-vectors to determine if our set of constraints can be satisfied. We leverage the Bit-Vector theory from the Cooperating Validity Checker (cvc5) SMT solver library \cite{cvc5}, where all relevant operations on bit-vectors (addition, subtraction, multiplication, and concatenation) are defined. Since these functions are native to the cvc5 library, we avoid the bottleneck of making external function calls when trying to automatically verify a circuit's correctness property. 

For a trivial example, we can use cvc5's Pythonic API to establish constraints in a small system, as shown in Figure \ref{fig-cvc5}. 
\begin{figure}
{\footnotesize
\begin{center}
    \begin{verbatimtab}
    from cvc5.pythonic import *
    a, b = BitVecs('a b', 5)
    x = BitVecVal(10, 5)
    y = BitVecVal(22, 5)
    z = BitVecVal(2, 5)
    // Bit-Vector representation of solve(a + 10 == 2 * b, b + 22 == 2 * a)
    solve(BVAdd(a, x) == BVMult(z, b), BVAdd(b, y) == BVMult(z, a))
    \end{verbatimtab}
\end{center}}
\caption{cvc5 Pythonic library example of SMT adapted from \cite{cvc5-example}}
\label{fig-cvc5}
\end{figure}
The \texttt{BitVecs('a b', 5)} syntax initializes two bit-vectors of size 5. \texttt{BitVecVal(10, 5)} initializes a bit-vector of size 5 and assigns it the value 10. \texttt{BVAdd(a, x)} performs addition on the bit-vectors \texttt{a} and \texttt{x}, and \texttt{BVMult(z, b)} multiplies the bit-vectors \texttt{z} and \texttt{b}. In addition to these operations, there is also \texttt{BVConcat(a,b)}, which concatenates bit-vector \texttt{b} onto the end (least significant side) of bit-vector \texttt{a} and is also present in the circuits used in this thesis. Note that bit-vectors of size 5 can represent integers up to $2^5 - 1$ or 31. The cvc5 library handles overflow of bit-vector operations by discarding the carry bit. 

On the seventh line in Figure \ref{fig-cvc5}, the constraints of the system are passed as parameters to the \texttt{solve()} call. The two restrictions state that adding 10 to \texttt{a} must be twice as much as \texttt{b}, and adding 22 to \texttt{b} must be twice \texttt{a}. Executing the script  will return the result \texttt{[a = 18, b = 14]}, a solution to the constraints, meaning that they are satisfiable.

The \textit{Overture/Prelude} framework has the ability to parse pre and postconditions of MPC protocols into similar constraints that the SMT solver can restrict the input and output space by. The definitions of these postconditions are what we test for satisfiability to automatically verify correctness. It is important to observe that the simple example returned one solution, but is not necessarily satisfiable for all combinations of \texttt{a} and \texttt{b}. In the SMT language, it is not possible to use universal quantifiers ($\forall$) to determine solutions. Instead, we follow a well-known theorem in SMT, also used in \cite{skalka-near-esop25}, which states:

\begin{center}
$\eqs_1 \models \eqs_2$ \textit{iff} $\eqs_1 \wedge \neg\eqs_2$ \textit{is not satisfiable.}
\end{center}

The $\models$ symbol indicates entailment, or that $\eqs_1$ models $\eqs_2$, meaning that all solutions for $\eqs_1$ are solutions for $\eqs_2$ as well. To connect it to our MPC protocols, $\eqs_1$ represents the precondition, and $\eqs_2$ represents the postcondition. Rather than checking all solutions for this, we look at the unsatisfiability of $\eqs_1 \wedge \neg\eqs_2$. If we verify the SMT solver over this condition and it does not satisfy, then there are no solutions for $\eqs_1$ that do not also satisfy $\eqs_2$. This is logically equivalent to the entailment. If it has no solutions, we can make the generalization that if the precondition holds for a solution, then the postcondition must also hold.

\section{\textit{Prelude} Language}
\label{sec:Prelude}
To introduce the \textit{Prelude} Language, we will start with a simple example. Suppose we have two parties, who have secrets $\elab{\secret{1}}{1}$ and $\elab{\secret{2}}{2}$, existing in the binary field $\mathbb{F}_2$. Our ideal functionality will be to compute the xor of the negation of ${\secret{1}}$ and ${\secret{2}}$, i.e., \texttt{not(}${\secret{1}\texttt{)}} + {\secret{2}}$. Figure \ref{fig-2-party-not-xor} illustrates the Boolean circuit diagram.
\begin{figure}
\centerline{\includegraphics[scale=.2]{2-party-not-xor.png}}
\caption{2-Party NOT XOR Circuit Diagram}
\label{fig-2-party-not-xor}
\end{figure}
This circuit is not secure, but we can replace the wire values and Boolean gates according to the GMW protocol. Using additive secret sharing, we can split each party's secret into two shares that appear random but xor together to reconstruct the secret, as shown in the \texttt{RECON} function. Recall that xoring singular bit values is the same as adding them. In the following \textit{Prelude} code in Figure \ref{fig-prelude-gmw}, it is assumed that the input variables for each gate are already shared in this way, as each function definition handles a wire's value as a combination of party 1's share and party 2's share. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
exprfunctions:

RECON(x) { (m[x]@1 + m[x]@2) }

not(x){ x + 1 }

cmdfunctions:

xorgmw(z:string, x:string, y:string) {
   m[z]@1 := (m[x] + m[y])@1; 
   m[z]@2 := (m[x] + m[y])@2
} 
postcondition: (RECON(z) == RECON(x) + RECON(y))

notgmw(z:string, x:string) {
   m[z]@1 := m[x]@1;
   m[z]@2 := not(m[x])@2
}
postcondition: ( RECON(z) == not(RECON(x)) )
\end{verbatimtab}
}
\caption{\textit{Prelude} libraries for GMW-style gates}
\label{fig-prelude-gmw}
\end{figure}
Another property of operations on bits in $\mathbb{F}_2$ is that we can find the  negation of a value $\mesg{x}$, made up of shares $\elab{\mesg{x}}{1}$ and $\elab{\mesg{x}}{2}$, by negating only one of the shares. This is utilized in the function definition of \texttt{notgmw(z,x)}. With these \textit{Prelude} functions, we can write the body of a main function that securely represents the circuit and realizes our ideal functionality in Figure \ref{fig-2-party-not-xor-code}. 
\begin{figure}
{\footnotesize
\begin{verbatimtab}
main() {
   notgmw("g1", "x");
   xorgmw("z", "g1", "y")
} 
postcondition: (RECON("z") == RECON("y") + not(RECON("x")))
\end{verbatimtab}
}
\caption{2-Party NOT XOR example protocol}
\label{fig-2-party-not-xor-code}
\end{figure}
We define the postcondition to assert the correctness property of the circuit. It states that the reconstruction of the output \texttt{"z"} should be equivalent to the sum of the reconstruction of input \texttt{"y"} with the negation of input \texttt{"x"}, which was our original intended functionality. This circuit verifies with an empty precondition, indicating it holds for all values of \texttt{"x"} and \texttt{"y"}. 

In collaboration with PhD Student Steven Baldasty and my advisor Dr. Christian Skalka, we also discussed and implemented changes to the \textit{Prelude} backend to include syntax for \texttt{BVAdd()}, \texttt{BVSub()}, \texttt{BVMult()}, and "\texttt{$\vert$}", the equivalent of \texttt{BVConcat()} in the cvc5 Bit-Vector SMT solver. This new syntax allows the tool to be used to verify circuits with bit-vector representations of numbers as inputs for addition, subtraction, and multiplication. There are two reasons for pivoting away from SMFF theory to the Bit-Vector theory. We believe that this library is more optimized and that, by having these functions natively defined in the cvc5 Bit-Vector library, we can call them more efficiently to perform operations on these larger circuits, reducing verification time. The other advantage is that it saves us the effort of redefining the ideal functionality for each operation, which would have been tedious. These functions can be invoked in the pre and postconditions of the larger circuits we tested and described below. 

\section{Methods}
\label{sec:Methods}
For an introduction to combining the new bit-vector syntax in \textit{Prelude} with an MPC protocol, we started with a 2-bit adder referenced from \cite{2-bit-adder}. The 2-bit adder, as shown in Figure \ref{fig-2-bit-adder-circuit} initially contained XOR, AND, and OR Boolean gates. Due to the cvc5 \texttt{BVAdd} definition dropping the carry bit, we were able to ignore the carry bit output wire and the corresponding gates. In addition to the \texttt{xorgmw(z,x,y)} \textit{Prelude} function defined in Figure \ref{fig-prelude-gmw}, we used supplemental libraries for an \texttt{andgmw(z,x,y)} function that utilizes oblivious transfer as defined in the GMW protocol. This definition has been omitted for simplicity.

\begin{figure}
    \centerline{\includegraphics[scale=0.2]{2-bit-adder.png}}
    \caption{2-Bit Adder circuit diagram adapted from \cite{2-bit-adder}}
    \label{fig-2-bit-adder-circuit}
\end{figure}
Without the carry bit output wire, there are only four relevant Boolean gates to translate into \textit{Prelude} code. The main function for the 2-bit adder in Figure \ref{fig-2-bit-adder} defines the secure version of the adder.
\begin{figure}
{\footnotesize
\begin{verbatimtab}
main() {
   xorgmw("z1", "x1", "y1"); // Bit 0 (least significant)
   andgmw("g1", "x1", "y1");
   xorgmw("g2", "x2", "y2");
   xorgmw("z2", "g2", "g1") // Bit 1 (most significant)
} 
postcondition: (|RECON("z2"), RECON("z1")| == BVAdd(|RECON("x2"), RECON("x1")|, 
                                                   |RECON("y2"), RECON("y1")|))
\end{verbatimtab}
}
\caption{2-Bit Adder in \textit{Prelude}}
\label{fig-2-bit-adder}
\end{figure}
Note that the postcondition explicitly states the correctness property. The output bit-vector, generated from the two output bits concatenated via "$\vert$", is equivalent to the sum of the two bit-vector inputs. The pattern of this postcondition is very similar to what we use in the larger, 64-bit circuits. Given the body of main and the postcondition, we used the \textit{Prelude} tool to pass the constraints to the SMT solver and automatically verify the protocol. As a small circuit, this verified quickly.

With a successful verification of a simple circuit using the Bit-Vector theory SMT solver, we decided to expand the evaluation to larger Boolean circuits, using the format of Bristol Circuits \cite{bristol}. Each line of the Bristol Circuit format includes the number of input wires, number of output wires, list of input and output wires, and the type of Boolean gate operation. For the circuits we wished to implement, it was not reasonable to manually translate hundreds or thousands of these lines into \textit{Prelude} code. 

The solution was to write a parsing script (see Appendix) in Perl. The script was made in collaboration with my advisor, Dr. Christian Skalka. Perl has a well-documented syntax for regular expressions, which was useful in parsing the Bristol Circuit format. For each line, we use the character class \textbackslash \texttt{w} to capture the gate operation. We could then go case by case for each gate and use the \textbackslash \texttt{d+} character class to retrieve the input and output wire values. These values are all that is necessary to generate a \textit{Prelude} function call, which was printed to the terminal for each gate. The script was designed to check for AND, INV (not), and XOR gates, which are present in various combinations in the larger circuits we wanted to test the tool on. The script was also extended to automatically generate the postcondition. The postcondition reversed the order of inputs to maintain the proper bit significance used in the cvc5 library, used \texttt{BVConcat} on both inputs and outputs, and the appropriate \texttt{BVAdd/BVSub/BVMult} operation.

The parser was run on Bristol Circuit text files for the 64-bit Adder, 64-bit Subtract, and 64-bit Multiplier circuits \cite{bristol}. It was then straightforward to populate the body of the main function with the list of GMW-style gate function calls and add the bit-vector operations to the corresponding postconditions. The implementations, ranging from hundreds to thousands of gates, are available in the \textit{Prelude} repository \cite{prelude-github}. When run with the \textit{Prelude} engine, the correctness properties for the addition and subtraction circuits are automatically verified after several seconds.

\section{Discussion}
From the table in Section \ref{sec:Results}, we can see that the 64-Bit Addition and 64-Bit Subtraction circuits verified in a very reasonable amount of time, both under 6 seconds. It does not take significantly longer than the 2 and 4 gate circuits. The 64-Bit Multiplication circuit, however, posed a challenge for the tool. The \textit{Prelude} process works in three parts. The parsing, the generation of constraints, and the verification with cvc5. Due to the circuit size of nearly 14,000 gates, the tool was unable to store all the constraints in memory before passing them along to the cvc5 solver. On the personal MacBook Air with the M2 chip, we ran into an OutOfMemoryError for Java heap space. In an effort to reduce the memory consumed by the constraints, we attempted to restrict the multiplication circuit inputs by writing a precondition such that each bit of the input bit-vectors was equal to 1 and a postcondition such that the least significant bit of the output bit-vector was also equal to 1. This restriction reduced the number of possible solutions to the set of constraints, but it did not result in a successful verification either. 

PhD Student Steven Baldasty ran the circuit on the Vermont Advanced Computing Center (VACC) machine, where it was still unable to get past the step of creating constraints for the cvc5 solver. This was with 90 gigabytes of memory allocated to the task. From this result, we believe that this is not necessarily a limitation of the cvc5 solver, but an opportunity to optimize the \textit{Prelude} backend to handle constraints more efficiently. Without generating all the constraints, we are unable to test if cvc5 can handle these larger circuits.

From Figure \ref{verification-graph}, we can see that the shape of the plotted verification times begins to hint at something roughly exponential. Even without any sample circuits between 127 and 376 gates, we have a good picture of how the verification time scales. If we were able to verify the 64-bit Multiplier circuit (13,675 gates) and some circuits in between, it would not be surprising to see the verification times continue to grow in this way. 

\section{Conclusions and Future Work}
By verifying circuits between 2 and 439 gates using cvc5 Bit-Vector theory, we have presented an empirical evaluation of the scalability of the \textit{Prelude} tool. The verification time appears to grow exponentially relative to the circuit size. We have not yet run into issues with how many constraints the cvc5 library can efficiently solve for, and it would be reasonable to assume that this growth would continue for the larger circuits.

Future work related to this thesis would involve looking for possible optimizations that can be made within the translation process for cvc5 constraints to more efficiently use memory and avoid the problem of running out of heap space. Currently, memory for the constraints is the bottleneck and we are unable to test if the verification of larger circuits would complete in a reasonable amount of time. If it is possible to refine the process for translating the function definitions into cvc5 constraints, Bristol Circuits also have a 64-Bit Signed Division circuit that contains nearly 30,000 gates and can serve as the next step up in scalability testing for automatic verification.

Additional work could involve extending this empirical evaluation of the Bit-Vector theory to Yao's Garbled Circuits protocols, which were considered in \cite{skalka-near-ppdp24, skalka-near-esop25}. Since the Garbled Circuits only require a constant number of rounds of communications with respect to the number of operations, it would be an interesting comparison against GMW, which scales linearly.
