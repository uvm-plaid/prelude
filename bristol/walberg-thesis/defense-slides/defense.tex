    \begin{itemize}
    \item Multi-Party Computation protocols perform secure computations on shared data across distributed participants.
    \item \textbf{Goal}: Ensure that the protocol is computing the correct value.
    \item \textbf{Solution}: Automatically verify it with \textit{Prelude} tool using Satisfiability Modulo Theories.

\begin{frame}{Contributions}
    Existing literature \footnote{Skalka and Near, ESOP 2025} evaluates the \textit{Prelude} framework on smaller circuits using Satisfiability Modulo Finite Fields theory. 
    \begin{itemize}
        \item Substantial verification time for small binary circuits.
        \item Relatively new compared to more established theories like the Satisfiability Modulo Bit-Vector theory.
    \end{itemize}
    Our contributions:
    \begin{itemize}
        \item Extension of the \textit{Prelude} backend to include \texttt{BVAdd/BVSub/BVMult/BVConcat}.
        \item Parsing script written in Perl to turn Boolean circuits into code that the \textit{Prelude} tool can verify.
        \item Secure Boolean circuits and their corresponding postconditions for Bit-Vector addition, subtraction, and multiplication.
        \item An empirical evaluation of the verification time as the circuit size grows.
    \end{itemize}
\end{frame}
\begin{frame}{Introduction to MPC}
    Multi-Party Computation (MPC) protocols serve as a way for distributed parties to compute shared functions on their data without revealing information about their data.
    \begin{itemize}
    \item \textbf{Objectives}: Compute the function correctly. Do not reveal \textit{too much} about each party's secrets.
    \item \textbf{Real-Ideal Paradigm}: Trusted oracle, ideal functionality, translation between ideal world and protocols in the real world.
    \end{itemize}
\end{frame}
\begin{frame}{Introduction to MPC}
    If an adversary is controlling a subset of the parties in a protocol, their goal might be to learn about the secrets of the honest parties. How do we stop them from learning anything they should not know?
    \begin{itemize}
    \item Conditional on the output, the adversary should be no better off in determining the secrets.
    \item If our protocol only gives them as much information as the ideal functionality output, then it is considered secure. This does not mean it is safe!
    \end{itemize}
    \textbf{Example:} Millionaires' Problem

\end{frame}
\begin{frame}{Introduction to MPC}
    We are considering the Goldreich-Micali-Wigderson (GMW) protocol, defined for binary circuits, where a party's input secret is represented by uniformly random values that xor together to form the secret. The values in these protocols are in the binary field $\mathbb{F}_2$ where $+$, $-$, and $*$ operations are defined.
    \begin{itemize}
    \item xor is equialent to $+$.
    \item $\wedge$ is equivalent to $*$.
    \item Since the field elements are finite, we can have uniform randomness.
    \item Values are \textit{local}, as parties are distributed. We communicate values between parties with messages. 
    \end{itemize}
\end{frame}

    \item $\flip{x}$ is a uniformly random sample from the field.
    \item $\rvl{x}$ is a public reveal.

\begin{frame}[fragile]{Satisfiability Modulo Bit-Vector Theory}
    Satisfiability Modulo Theories (SMT) are used for the problem of finding solutions to satisfy a set of constraints.
    % an extension of the Boolean Satisfiability Problem where we search to find a set of inputs such that some set of constraints is satisfiable.
    \begin{itemize}
        \item This is done with respect to a theory, in our case Bit-Vector theory.
        % \item A SMT solver automatically searches for a solution for a subset of Bit-Vectors.
    \end{itemize}
    We used the Cooperating Validity Checker (cvc5) library, which has an established Bit-Vector theory for SMT solving.
    \begin{itemize}
    \item All bit-vector operations are defined natively, no bottleneck to make external calls.
    % \item Functionality of each operation already exists and we avoid writing them ourselves.
    \end{itemize}
    \begin{figure}
        {\footnotesize
        \begin{center} 
            \begin{verbatimtab}
from cvc5.pythonic import *
a, b = BitVecs('a b', 5)
x = BitVecVal(10, 5)
y = BitVecVal(22, 5)
z = BitVecVal(2, 5)
// BV representation of solve(a + 10 == 2 * b, b + 22 == 2 * a)
solve(BVAdd(a, x) == BVMult(z, b), BVAdd(b, y) == BVMult(z, a))
        \end{verbatimtab}
        \end{center}
        }
        \caption{cvc5 Pythonic library example of SMT}
        \label{fig-cvc5}
    \end{figure}
\end{frame}
\begin{frame}{Satisfiability}
    \begin{figure}
        \begin{center}
    {\footnotesize{\texttt{solve(BVAdd(a, x) == BVMult(z, b), BVAdd(b, y) == BVMult(z, a))}}}
        \end{center}
    \end{figure}
    With the solution \texttt{[a = 18, b = 14]}, this set of constraints is satisfiable. But, this is not good enough for verifying a correctness property for an MPC protocol.
    \begin{itemize}
        \item We want to ensure it is satisfied in \textit{all} cases, but SMT solvers often disallow quantifiers.
    \item 
        \textbf{Theorem: }$\eqs_1 \models \eqs_2$ \textit{iff} $\eqs_1 \wedge \neg\eqs_2$ \textit{is not satisfiable.}
        \begin{itemize}
            \item $\models$ means that $\eqs_1$ \textit{models} $\eqs_2$.
            \item Say $\eqs_1$ is the precondition and $\eqs_2$ is the postcondition.
            \item If the negation of the postcondition holds for any solution of the precondition, the protocol is incorrect!
        \end{itemize}
    \end{itemize}
\end{frame}
\begin{frame}[fragile]{\textit{Prelude} Language}
    We can execute a \textit{Prelude} program to \textit{generate} an \textit{Overture} MPC protocol.
    \begin{figure}
        \begin{minipage}[t]{.5\textwidth}
            \begin{itemize}
                \item This gives us the ability to define pre and postconditions for our GMW-style functions.
                \item With secure versions of \textit{xor} and \textit{not}, we can define secure binary circuits.
            \end{itemize}
            \end{minipage}%
            \begin{minipage}[t]{.5\textwidth}
                    {\scriptsize

cmdfunctions:

xorgmw(z:string, x:string, y:string) {
    m[z]@1 := (m[x] + m[y])@1; 
    m[z]@2 := (m[x] + m[y])@2
} 
postcondition: (RECON(z) == 
                RECON(x) + RECON(y))

notgmw(z:string, x:string) {
    m[z]@1 := m[x]@1;
    m[z]@2 := not(m[x])@2

postcondition: ( RECON(z) == 
                not(RECON(x)) )
                    \end{verbatimtab}

\begin{frame}{Conclusions and Future Work}
    How did the \textit{Prelude} automatic verification engine perform?
    \begin{itemize}
        \item 64-bit Addition, 64-bit Subtraction verify reasonably.
        \item 64-bit Multiplication posed significant challenge to the tool.
    \end{itemize}

    \item Extend evaluation to Yao's Garbled Circuits, which were considered in \footnote{Skalka and Near, ESOP 2025}.
